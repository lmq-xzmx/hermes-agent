# Hermes File Manager 系统架构文档

## 领域分类

**领域**：企业级文件管理 / 云存储协作平台 (Enterprise File Management & Cloud Storage Collaboration Platform)

**子领域**：
- 多租户存储资源管理 (Multi-tenant Storage Resource Management)
- 基于角色的访问控制 (RBAC - Role-Based Access Control)
- 团队协作与空间隔离 (Team Collaboration & Space Isolation)

---

## 系统架构概览

```
StoragePool (存储池)
    └── Space (空间) [type: root/team/private]
            └── SpaceMember (成员关系)
                    └── User (用户)
```

---

## 1、新系统初始化步骤

```
1. 启动 server.py
   ↓
2. TeamService.ensure_default_pool() 创建默认存储池
   ↓
3. 创建 admin 用户 (默认管理员)
   ↓
4. 内置角色初始化 (admin/editor/viewer/guest)
   ↓
5. FileService.register_pool() 注册存储引擎
```

### 存储结构

```
{storage_root}/
├── pools/           # 存储池根目录
├── teams/           # 团队目录 (Team = Space)
│   └── {team_id}/
│       └── members/
│           └── {user_id}/
└── spaces/          # 空间目录
    └── {space_id}/
        ├── shared/
        └── members/ (仅 private 类型)
```

---

## 2、新用户注册后的正确使用步骤

```
1. 用户注册 → 创建 User 记录 (role: viewer 默认)
   ↓
2. 用户登录 → 获取 JWT token
   ↓
3. 用户需要加入团队/空间才能使用存储
   ↓
4. 通过两种方式加入：
   - 管理员邀请 → SpaceMember 创建
   - 使用邀请码 → SpaceCredential 验证后加入
   ↓
5. 成为成员后获得该空间的文件访问权限
```

### 权限获取流程

```
User → SpaceMember (space_id + role) → 获取 Space 权限
```

---

## 3、新团队创建后的正确使用步骤

```
1. 管理员/用户调用 POST /api/v1/teams
   参数: name, storage_pool_id, max_bytes (可选)
   ↓
2. 系统创建 Team (即 Space, type="team")
   - 生成唯一 team_id
   - 创建 team root 目录: {pool_base_path}/teams/{team_id}
   - 创建 members 子目录
   ↓
3. 创建者自动成为 TeamMember (role: owner)
   ↓
4. 团队 Owner 可以：
   - 邀请新成员 (通过 SpaceCredential)
   - 分配配额 (max_bytes)
   - 上传/管理共享文件
```

### 团队文件路径

```
{pool_base_path}/teams/{team_id}/members/{user_id}/  ← 各成员个人目录
{pool_base_path}/teams/{team_id}/shared/              ← 团队共享文件
```

---

## 4、新空间创建后与团队、用户的关系

Space 创建时需要指定：
- `storage_pool_id` (绑定存储池)
- `owner_id` (创建者)
- `space_type`: "root" | "team" | "private"

### 关系链

```
Space (space_type)
├── "root"   → 绑定存储池，无父空间
├── "team"   → 可有父空间 (root space)，团队共享
└── "private" → 子空间，需要通过 SpaceRequest 申请创建
```

### 空间层级示例

```
Root Space (pool绑定)
  └── Team Space A (团队A成员共享)
        ├── Private Space (成员A的个人空间)
        └── Private Space (成员B的个人空间)
```

---

## 5、存储池创建后如何分配给不同团队和个人

```
1. Admin 调用 POST /api/v1/pools
   参数: name, base_path, protocol, total_bytes
   ↓
2. StoragePool 记录创建
   ↓
3. 创建 Team/Space 时指定 storage_pool_id
   ↓
4. 每个 Team 独立配额 (max_bytes)
   ↓
5. 写入文件时检查：
   - Team 配额是否超限
   - Pool 剩余空间是否足够
```

### 配额检查流程

```
文件上传 → check_quota_for_write(space_id, size)
  → 检查 Space.max_bytes
  → 检查 Pool.free_bytes
  → 足够则写入，否则抛出 QuotaExceeded
```

---

## 6、私人空间与团队空间之间的关系

### 私人空间 (Private Space) 创建流程

```
1. 用户发起 SpaceRequest (请求创建私人空间)
2. Space Owner 审核请求
3. 审核通过 → 创建 Private Space
   - parent_id = 团队 Space
   - space_type = "private"
   - owner_id = 申请人
   ↓
4. 私人空间结构：
   {pool_base_path}/spaces/{private_space_id}/
   ├── shared/         ← (团队不可见)
   └── members/{user_id}/  ← 仅所有者可见
```

### 权限隔离

- 团队成员无法访问其他成员的 Private Space
- Private Space Owner 可邀请他人协作

---

## 7、管理员账号与普通用户之间的关系

### Role 系统

| Role   | 权限                                          |
|--------|-----------------------------------------------|
| admin  | 所有权限 + 用户管理 + 存储池管理              |
| editor | 读写指定路径，删除/管理受限制                  |
| viewer | 只读指定路径                                   |
| guest  | 最小权限，仅读共享资源                         |

### 用户角色分配

```
User.role_id → Role.id → PermissionRule (路径模式匹配)
```

### Admin 特有权限

- 创建/删除 StoragePool
- 创建/删除 Space
- 分配用户角色
- 查看所有审计日志

### PermissionRule 路径匹配

```
path_pattern: "/**"           → 匹配所有路径
permissions: "read,write,list,delete"
priority: 0                   → 多规则时优先级
```

```
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      用户注册/登录                               │
├─────────────────────────────────────────────────────────────────┤
│  Register → User(role=viewer)                                   │
│  Login → JWT Token                                               │
│  Join Space → SpaceMember(role=member)                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      团队/空间创建                                │
├─────────────────────────────────────────────────────────────────┤
│  Create Team → Space(type=team, owner=user)                     │
│  Create SpaceRequest → 待审核                                    │
│  Approve → Space(type=private)                                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      文件操作流程                                │
├─────────────────────────────────────────────────────────────────┤
│  Upload → 检查配额 → 写入存储 → 更新 used_bytes                  │
│  Delete → 软删除 (DeletedFile) → 回收站                          │
└─────────────────────────────────────────────────────────────────┘
```

## 生命周期操作约束

### 约束规则

| 操作 | 前置条件 | 违反时提示 |
|------|---------|-----------|
| 上传文件 | 必须是 Space 成员 | "请先加入团队或空间才能上传文件" |
| 创建团队 | 已登录 + 有可用存储池 | "请先联系管理员创建存储池" |
| 创建私人空间 | 已是团队 Space 成员 | "请先加入团队，再申请私人空间" |
| 删除存储池 | 没有任何团队使用此池 | "该存储池仍有团队使用，无法删除" |
| 邀请成员 | 是团队 Owner | "只有团队所有者可以邀请新成员" |
